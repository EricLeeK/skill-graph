市面上 34,396 个 Agent Skill 的描述严重语义不完备：它们用"功能语言"写就，而用户用"任务语言"查询。我们让 LLM 为每个 Skill 生成 10 条真实用户查询，构建 Multi-Vector 索引。预处理阶段一次烧 Token，运行时仅需向量运算——零 Token、22ms、71.0% Recall@10，超过 UCSB Agentic 基线 68.3%。

问题：描述语义不完备

Skill 描述存在三项结构性缺陷：

  • 功能导向而非任务导向：描述讲"做什么"，不讲"解决什么问题"
  • 模板化同质：大量 Skill 描述结构雷同，覆盖的用户表达极窄
  • 缺失使用场景："新成员加入时"、" hackathon demo 前"等关键上下文完全空白

方案：SynQ 语义重构

为每个 Skill 生成 10 条多样化合成查询：

  • 覆盖 10 种语用模式：直接请求、问题陈述、比较式、场景化、口语化、紧急化等
  • Multi-Vector 索引：Skill 描述 embedding + 10 条 SynQ embedding
  • 检索评分：max(skill_sim, best_syn_sim)，兼顾技术用户与 casual 用户

两阶段架构预处理投资 → 运行时零成本

阶段一 · 预处理
LLM 为 34K Skill 各生成 10 条 SynQ，嵌入后存入 Multi-Vector 索引。一次性投入 ~765M Token（~$765），此后永久复用。
阶段二 · 运行时
Query embedding 与索引中全部 34K Skill 做向量点积，取 Top-10 返回。无 LLM 调用，无 API 成本，延迟仅 22ms。
市场视角：语义重构应由 Skill 市场平台方（如 OpenAI GPT Store、GitHub Registry）统一完成，而非单个 Agent 开发者。原因：所有 34K Skill 的描述缺陷是系统性而非个案性的；平台一次性投入的正外部性覆盖所有下游 Agent。
