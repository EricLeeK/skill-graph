01 / Problem

The Skill Discovery Gap

Modern LLM agents are equipped with thousands of reusable skills—yet finding the right skill for a given task remains a critical bottleneck. The gap is not in capability, but in discovery.

Agent skills are standardized knowledge artifacts: markdown files containing API guides, code templates, detection thresholds, and best practices. A single skill ecosystem (skills.sh) hosts over 34,000 such artifacts, spanning domains from network security to financial modeling.

Yet when an agent faces a real-world task—"analyze this PCAP file for intrusion patterns"—it must somehow locate the single skill (pcap-analysis) among 34,000 candidates, based solely on the user's natural language query.

The query "analyze network traffic" and the skill name "pcap-analysis" live in different semantic spaces. This is the query-skill semantic gap.

SkillsBench (UCSB et al., 2026) provides 87 curated real-world tasks with ground-truth skill requirements. On this benchmark, the best agentic retrieval system achieves 68.3% Recall@10—but at the cost of ~5,000 tokens and multiple LLM calls per query.

02 / Core Insight

Intelligence Belongs in the Index, Not the Runtime

The key realization: we can afford to use LLMs during preprocessing—once per skill—but we cannot afford LLM calls at query time.

"What if, instead of asking an LLM to understand skills at query time, we ask it to understand them once—during index construction—and embed that understanding directly into the retrieval index?" — Central Design Principle of SkillGraph

For each of the 34,000 skills, we use an LLM (Kimi k1.5) to generate 10 diverse synthetic user queries—the kinds of natural language questions that would lead a user to this skill.

For pcap-analysis, the LLM generates queries like:

"analyze network packet captures" · "detect port scanning in traffic logs" · "compute network statistics from pcap files"

These synthetic queries are embedded and stored alongside the skill's own embedding, creating a multi-vector representation of each skill. At runtime, retrieval becomes a simple vector similarity search—zero tokens, millisecond latency.

05 / Ablation

Component Analysis

Understanding the contribution of each component through controlled ablation on SkillsBench.

Component Contribution

Per-Fold Recall@10 (Dense + SynQ)

Synthetic Query Generation (SynQ) is the single most impactful component, providing +9.3pp improvement over the dense baseline. This confirms the central hypothesis: bridging the query-skill semantic gap at index time is far more efficient than attempting to bridge it at runtime.

SACR provides a modest speedup (2.8x faster than dense) but introduces a recall penalty of ~3pp. The optimal SACR configuration varies across folds (L0=12-24, L1=60-100, L2=150-250), suggesting that community granularity should be task-dependent.

The high variance (±16.2%) across folds reveals an important insight: SkillsBench tasks are not uniformly difficult. Some folds contain tasks with highly generic queries (e.g., "fix build errors") where GT skills have no lexical overlap with the query, making semantic retrieval challenging regardless of method.

06 / Conclusion

What We Learned

SkillGraph demonstrates that preprocessing intelligence can substitute for runtime intelligence. By using LLMs to deeply understand skills during index construction—generating synthetic queries, building multi-vector representations—we achieve retrieval performance that exceeds state-of-the-art agentic methods while eliminating all runtime LLM calls.

The multi-vector index (skill embedding + synthetic query embeddings) is the core innovation. It transforms the retrieval problem from "match query to skill name" to "match query to user intent"—a fundamentally easier problem because the synthetic queries are generated from the same distribution as real user queries.

Speed and accuracy are not contradictory. At 17.5ms per query and 72.1% Recall@10, SkillGraph surpasses the UCSB agentic baseline (68.3%) by 3.8pp while being ~170x faster. For latency-sensitive agent systems, this is transformative.

Key Takeaway: The future of agent skill retrieval lies not in smarter runtime reasoning, but in smarter index construction. Every token spent during preprocessing pays dividends at runtime—in latency, cost, and reliability.

07 / Future Work

What Comes Next

Near-term

Graph RAG Expansion

Skills are connected by semantic relationships (related_skill_ids). A Graph RAG approach—starting from top-K retrieved skills and traversing the relationship graph—could surface relevant skills that were missed by vector similarity alone, particularly for multi-skill tasks.

Near-term

SACR v2: Graph Attention Routing

Current SACR uses community centroids for routing. Replacing this with a learned graph attention mechanism—where community relevance is conditioned on the query embedding through attention weights—could reduce the recall gap between SACR and dense retrieval from 3pp to near-zero.

Mid-term

Adaptive Synthetic Queries

Current synthetic queries are static. An adaptive approach—monitoring which synthetic queries actually retrieve their parent skill on real user queries, and re-generating underperforming ones—could continuously improve index quality without human intervention.

Long-term

Unified Skill Agent

The ultimate vision: a retrieval module so fast and accurate that it becomes invisible. The agent simply states its intent, and the right skills are loaded automatically—no explicit retrieval step, no latency perceived by the user. SkillGraph is a step toward that vision.
